# 🤝 Partner Mode — архитектура (Фаза 7)

> Решение на собственика (2026-06-12): backend **Supabase** (EU регион,
> безплатен тир за старт). Желязното правило на фазата: **сървърът
> никога не вижда съдържание** — само шифровани блокчета и минимални
> метаданни за маршрутизация.

## Принципи

1. **Zero-knowledge**: цялото криптиране/декриптиране е на клиента.
   В Supabase лежат само ciphertext, nonce и MAC. Без ключове, без
   заглавия, без таговe — нищо четимо.
2. **Granular sharing**: нищо не се споделя по подразбиране. Всяко
   споделено нещо е изричен избор („сподели това с Н.").
3. **Изключването е едностранно и моментално**: всяка от страните
   може да прекъсне връзката; локалните данни на другия не се отнемат
   (както при иззета бележка на хартия), но потокът спира веднага.
4. **Без INTERNET permission до тази фаза** — добавя се чак когато
   Partner Mode влезе в билда, и мрежата се ползва само за sync.

## Криптографска схема

| Стъпка | Алгоритъм | Защо |
|---|---|---|
| Ключове на устройството | X25519 (Curve25519 ECDH) | Индустриален стандарт (Signal, WireGuard) |
| Общ секрет на двойката | ECDH(myPrivate, theirPublic) → HKDF-SHA256 | И двете страни го изчисляват сами; не пътува по мрежата |
| Шифроване на записи | AES-256-GCM (random nonce на запис) | AEAD: поверителност + интегритет; хардуерно ускорено |
| Визуална проверка | SAS: 2 емоджита от SHA-256(споделен ключ) | Двамата сравняват на глас → MITM защита |

Имплементация: пакет `cryptography` (pure Dart → host тестове без
устройство). Ключът на двойката се пази само в OS Keystore
(SecureStore), никога в базата или бекъпите.

## Сдвояване (pairing)

```
Ана (кани)                        Сървър                Борис (приема)
1. генерира X25519 двойка
2. POST pairing {code, pubA}  →  пази code+pubA
3. казва кода на Борис (на живо/чат — кодът сам по себе си не стига:
   нужен е и pubB обмен, а SAS проверката хваща подмяна)
                                                  4. GET pairing(code) → pubA
                                                  5. генерира двойка, POST pubB
6. realtime: pubB пристига
7. shared = ECDH+HKDF              (вижда само pубличните ключове)    7'. shared = ECDH+HKDF
8. 💜 SAS емоджита се показват и на двата екрана → сравняват на глас
9. потвърждение → pairing row се трие; остава couple_id
```

- Кодът е 8 знака от недвусмислена азбука (без 0/O, 1/I/l), живот
  15 минути, еднократен.
- След сдвояване сървърът знае само: две анонимни device id-та са
  двойка `couple_id`. Без имена, имейли, телефони (анонимен Supabase
  auth).

## Модел на данните (Supabase)

```sql
pairings(code text pk, pub_a bytea, pub_b bytea, created_at, expires_at)
couples(id uuid pk, member_a uuid, member_b uuid, created_at)
shared_items(id uuid pk, couple_id fk, author uuid,
             kind smallint,            -- 0 запис, 1 ден, 2 nudge…
             cipher bytea, nonce bytea, mac bytea,   -- само шифровано
             created_at, deleted_at)
```

RLS (row level security): четеш/пишеш само редове на собствения си
`couple_id`. `kind` е единствената нешифрована семантика — нужна за
ефективен pull; не издава съдържание.

## Клиентска архитектура

```
lib/partner/
  couple_crypto.dart      ← ключове, ECDH, AES-GCM, SAS (НЯМА мрежа)
  sync_backend.dart       ← абстракция + InMemorySyncBackend (тестове)
  supabase_backend.dart   ← реалната имплементация (чака проекта)
  partner_repository.dart ← състояние: none → inviting → linked;
                            persist в SecureStore; granular share API
```

Всичко освен `supabase_backend.dart` работи и се тества офлайн —
mock-ът симулира сървъра в паметта, тестовете доказват, че по
„жицата" минава само ciphertext.

## Какво се споделя във v1 (вертикален срез)

1. **Споделена бележка** („за нас") — най-простият запис.
2. **Споделен календарен маркер** — само избрани дни/типове, изрично.
3. **„Мисля за теб" nudge** — еднобайтово съобщение, пак шифровано.

Couple Plan монетизацията и Resonance (Фаза 7.5) стъпват върху същия
канал.
